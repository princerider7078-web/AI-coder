SECTION 01 — DOCUMENT OVERVIEW & CONVENTIONS
================================================================================

1.1 PURPOSE
-----------
This Product Requirements Document (PRD) defines the complete functional and
non-functional requirements for the GROWPLANTS platform. It is intended as the
single source of truth for:
  - Frontend developers (Web & Mobile)
  - Backend developers (API & Database)
  - AI coding tools (Cursor, GitHub Copilot, Claude, etc.)
  - QA engineers
  - DevOps engineers

This document translates the Business Requirements Document (BRD v1.0) into
precise, implementable specifications. Every feature described here maps to a
business goal. No feature in this document exists without a BRD justification.

1.2 DOCUMENT CONVENTIONS
--------------------------
- [REQUIRED]   = Must be built in Phase 1 (MVP Launch)
- [PHASE-2]    = Required for Haryana expansion
- [PHASE-3]    = Required for national expansion
- [FUTURE-AI]  = AI-powered features in future roadmap
- [OPTIONAL]   = Nice-to-have, build if time permits
- FR-XXX       = Functional Requirement ID
- NFR-XXX      = Non-Functional Requirement ID
- API-XXX      = API Endpoint ID
- DB-XXX       = Database Table/Model ID
- UI-XXX       = UI Screen/Component ID

1.3 ASSUMPTIONS
---------------
1. Platform launches in Sonipat, Haryana first.
2. Business owner manages inventory directly (no multi-vendor product marketplace).
3. Service providers are third-party freelancers onboarded by admin.
4. Payment gateway: Razorpay (India-first, supports UPI, cards, wallets).
5. Delivery is managed in-house or via local courier for Phase 1.
6. Mobile app: React Native (single codebase, Android + iOS).
7. Backend: Node.js + Express or NestJS (developer's choice, must be RESTful).
8. Database: PostgreSQL (primary) + Redis (caching/sessions).
9. File storage: AWS S3 or Cloudflare R2.
10. All prices in Indian Rupees (INR).
11. Language: Hindi + English bilingual support (UI labels bilingual).
12. Guest checkout is NOT allowed — account required for all purchases.

1.4 OUT OF SCOPE (Phase 1)
---------------------------
- Multi-vendor product marketplace (sellers listing their own products)
- International shipping
- Cryptocurrency payments
- AR plant preview (future)
- AI plant doctor / recommendations (future)
- Loyalty points / rewards program
- Affiliate marketing module
- Live chat with gardeners

SECTION 02 — PRODUCT VISION & SCOPE
================================================================================

2.1 PRODUCT OVERVIEW
---------------------
GROWPLANTS is a location-based eCommerce and Service Marketplace platform for
the gardening ecosystem in India. It combines:

  PILLAR 1: PRODUCT STORE
    → Online store for plants, planters, gardening supplies.
    → Owner-managed inventory.
    → Delivery within serviceable pincodes.

  PILLAR 2: SERVICE MARKETPLACE
    → Customers can book gardening services.
    → Services fulfilled by verified service providers.
    → Admin approves all service providers.
    → Commission-based revenue model.

  PILLAR 3: ADMIN CONTROL CENTER
    → Full platform management by business owner/admin.
    → Order, booking, inventory, provider, and customer management.

2.2 PHASE SCOPE
----------------
PHASE 1 (MVP — 0-6 months):
  - Customer Web App + Mobile App
  - Product Catalog (Plants, Planters, Gardening Products)
  - Cart + Checkout + Payment
  - Order Management
  - Service Booking (basic)
  - Service Provider Panel (basic)
  - Admin Panel (full)
  - Inventory Management
  - Review & Rating
  - Notification System (Email + Push + SMS)
  - Sonipat city only

PHASE 2 (6-18 months):
  - Multi-city support (Haryana)
  - Advanced analytics dashboard
  - Featured listings for service providers
  - Vendor subscription model

PHASE 3 (18-36 months):
  - National expansion
  - AI recommendation engine
  - AI plant doctor
  - AR plant preview
  - Advanced B2B features

2.3 PLATFORM TYPES
-------------------
1. Customer Web App       → React.js (responsive, works on mobile browsers too)
2. Customer Mobile App    → React Native (Android + iOS)
3. Admin Web Panel        → React.js (desktop-first, separate app)
4. Service Provider App   → React Native (shared with customer app, role-based)
5. Backend API            → Node.js / NestJS (REST API)
6. Database               → PostgreSQL + Redis
7. File Storage           → AWS S3 / Cloudflare R2

SECTION 03 — SYSTEM ARCHITECTURE OVERVIEW
================================================================================

3.1 HIGH-LEVEL ARCHITECTURE
-----------------------------

  [Customer Web/Mobile App]
           |
           | HTTPS REST API
           ↓
  [API Gateway / Load Balancer]
           |
           ↓
  [Backend API Server (Node.js)]
     |          |          |
     ↓          ↓          ↓
[PostgreSQL]  [Redis]   [AWS S3]
(main DB)   (cache/    (images/
             sessions)  files)
     |
     ↓
[Razorpay API]  [Firebase (Push)]  [Twilio/MSG91 (SMS)]  [SendGrid (Email)]

3.2 MODULE BREAKDOWN
---------------------
Backend modules (separate services or route groups):
  - auth           → Authentication, sessions, OTP
  - users          → Customer profiles, addresses
  - products       → Catalog, categories, variants
  - inventory      → Stock management
  - cart           → Shopping cart (Redis-backed)
  - orders         → Order lifecycle management
  - services       → Service catalog
  - bookings       → Booking lifecycle
  - providers      → Service provider profiles
  - payments       → Razorpay integration
  - reviews        → Reviews and ratings
  - notifications  → Email, SMS, push
  - admin          → Admin-only endpoints
  - search         → Search and filter
  - locations      → City, pincode, area management
  - uploads        → File/image upload to S3
  - returns        → Return and refund management
  - reports        → Analytics and reporting

3.3 DATA FLOW: PRODUCT ORDER
-----------------------------
Customer → Browse Products → Add to Cart → Enter Address
→ Select Delivery Slot (if applicable) → Review Order
→ Payment (Razorpay) → Payment Webhook → Order Created
→ Admin Notified → Admin Confirms → Delivery Assigned
→ Customer Notified → Delivered → Order Completed
→ Review Prompt Sent

3.4 DATA FLOW: SERVICE BOOKING
--------------------------------
Customer → Browse Services → Select Service
→ Enter Location (coordinates captured) → Select Date/Time
→ Select Service Provider (or auto-assign) → Review & Confirm
→ Payment (advance or COD) → Booking Created
→ Provider Notified → Provider Confirms → Service Delivered
→ Booking Completed → Review Prompt Sent

SECTION 04 — TECH STACK RECOMMENDATIONS
================================================================================

4.1 FRONTEND (Customer Web)
-----------------------------
Framework        : React.js 18+
State Management : Redux Toolkit + RTK Query (for API calls)
Styling          : Tailwind CSS + shadcn/ui component library
Routing          : React Router v6
Forms            : React Hook Form + Zod (validation)
Maps             : Google Maps API (location picker, delivery area)
Payments         : Razorpay Web SDK
SEO              : React Helmet / Next.js (consider Next.js for SEO benefits)
Image Handling   : React-image-crop (for uploads), Cloudinary/S3 URLs
Build Tool       : Vite or Next.js
PWA              : Yes (add to home screen, offline capability for basic browsing)

RECOMMENDATION: Use Next.js 14+ (App Router) for better SEO, server-side
rendering, and image optimization — critical for a product eCommerce site.

4.2 MOBILE APP
---------------
Framework     : React Native (Expo managed workflow)
Navigation    : React Navigation v6
State         : Redux Toolkit
Payments      : Razorpay React Native SDK
Maps          : react-native-maps + Google Maps
Push Notifs   : Firebase Cloud Messaging (FCM)
Storage       : AsyncStorage (local), Redux Persist
Camera        : Expo Camera (for review photos, plant photos)
Platform      : Android 8.0+ and iOS 13+

4.3 BACKEND API
----------------
Runtime       : Node.js 20 LTS
Framework     : NestJS (TypeScript, modular, enterprise-grade)
              OR Express.js + TypeScript (simpler, faster to build)
ORM           : Prisma (PostgreSQL)
Auth          : JWT (access token 15min) + Refresh Token (7 days) + HTTP-only cookies
Validation    : Joi or class-validator (NestJS)
File Upload   : Multer → AWS S3 SDK
Caching       : Redis (ioredis)
Queues        : Bull (Redis-based) for background jobs (emails, notifications)
Logging       : Winston + Morgan
API Docs      : Swagger / OpenAPI 3.0

4.4 DATABASE
-------------
Primary DB    : PostgreSQL 15+
Cache / Queue : Redis 7+
ORM           : Prisma
Migrations    : Prisma Migrate
Search        : PostgreSQL Full-Text Search (Phase 1), Elasticsearch (Phase 3)

4.5 INFRASTRUCTURE
-------------------
Cloud         : AWS / DigitalOcean / Railway (Phase 1 — cost-effective)
Web Server    : Nginx (reverse proxy)
SSL           : Let's Encrypt / AWS Certificate Manager
CDN           : Cloudflare
File Storage  : AWS S3 / Cloudflare R2
Domain        : growplants.in (recommended)
Email         : SendGrid or AWS SES
SMS           : MSG91 or Textlocal (India-specific, cheap)
Push Notifs   : Firebase Cloud Messaging (free tier sufficient for Phase 1)
Monitoring    : PM2 (process manager) + Sentry (error tracking)

4.6 DEVELOPMENT TOOLS
----------------------
Version Control : Git + GitHub / GitLab
CI/CD           : GitHub Actions

API Testing     : Postman / Thunder Client
Code Quality    : ESLint + Prettier + Husky (pre-commit hooks)

This enables shareable filtered URLs and proper browser back/forward behavior.

Filter parameters (products):
  category_id     → UUID (multi-select)
  min_price       → integer
  max_price       → integer
  min_rating      → 1 to 5
  suitable_for    → indoor | outdoor | balcony | office (multi-select)
  sunlight        → full_sun | partial_shade | shade | indirect (multi-select)
  difficulty      → easy | moderate | expert (multi-select)
  in_stock        → boolean
  tags            → array

Filter parameters (services):
  category_id     → UUID (multi-select)
  min_price       → integer
  max_price       → integer
  min_rating      → 1 to 5
  city            → city name
